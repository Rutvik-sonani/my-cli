# Security Policy

## Reporting a Vulnerability

If you discover a security vulnerability in **MyCLI** (the CLI, generators, or official packages), please report it responsibly.

**Do not** open a public GitHub issue for security-sensitive reports.

Preferred channels:

1. **GitHub Security Advisory** — use [Report a vulnerability](https://github.com/mycli/mycli/security/advisories/new) on the repository (adjust URL if the canonical repo differs).
2. **Private contact** — email the maintainers if GitHub advisories are unavailable.

Include:

- Affected version(s)
- Steps to reproduce
- Impact assessment
- Suggested fix (if any)

We aim to acknowledge reports within **72 hours** and provide a remediation timeline when possible.

## Supported Versions

| Version | Supported |
|---------|-----------|
| Latest release | ✅ |
| Previous major | ⚠️ Best effort |
| Older | ❌ |

## Scope

**In scope:** `@mycli/cli`, packages under `packages/`, official plugins under `plugins/official/`, and template output that ships with the CLI.

**Out of scope:** Applications generated by MyCLI (user-owned code), third-party community plugins unless explicitly maintained by the core team, and infrastructure you deploy yourself.

## Generated Projects

MyCLI scaffolds security helpers (`my security audit`, `my security scan-secrets`, Helmet/CORS templates). Generated apps remain the responsibility of their owners — run dependency audits and secret scans in CI for production deployments.

## Safe Harbor

We support good-faith security research. Do not access user data without permission, perform destructive testing on shared infrastructure, or disclose issues publicly before a fix is available.
